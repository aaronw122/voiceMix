I would pick **two repos**: one for the iOS app, and one for the full-stack web app. That matches the real ownership and release boundaries: the Swift/iMessage app has its own build system, distribution path, platform constraints, and deployment cadence, while the web frontend and backend will likely share TypeScript types, auth/session assumptions, API contracts, tooling, environment config, and CI workflows.

The main thing you give up versus a monorepo is atomic cross-platform change management. If a product feature requires coordinated iOS, frontend, and backend changes, you will need versioned APIs, compatibility discipline, and sometimes linked PRs instead of one commit that updates everything. That is a real cost, but usually smaller than forcing iOS and web tooling into one repo before the team or product complexity demands it.

I would choose a monorepo if you expect one small team to constantly change iOS and web together, you want shared generated clients/contracts as a first-class workflow, and you are willing to invest early in clean CI boundaries. I would choose three repos only if frontend and backend have clearly separate teams, deploy independently, and do not benefit much from shared TypeScript contracts. For an early product, **iOS repo + full-stack web repo** is the most pragmatic default.
